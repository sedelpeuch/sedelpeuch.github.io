Exemple des expressions arithmétiques​

Une grammaire reconnaissant les expressions arithmétiques se reconnaît avec les expressions suivantes

  1. exp -> ID
  2. exp -> Cste
  3. exp -> exp + exp
  4. exp -> exp * exp
  5. exp -> (exp)

Ces 5 règles sont appelées les règle de grammaire ou règle de production. Dans ces règles on distingue :

  • les terminaux T=ID,CSTE,+,\*,(,)T={ID,CSTE,+,\*,(,)}T=ID,CSTE,+,\*,(,) ce sont les unités lexicales (aussi appelées tokens) qui sont données en entrée de l'analyseur. C'est un dire, c'est une suite / un mot de terminaux, donc dans T∗T*T∗
  • les non-terminaux N={exp}N=\{exp\}N={exp}, ce sont les ensembles de mots de terminaux qu'on souhaite construire ou définir
  • les règles de production sont : qui peuvent être lue comme des inclusion ensemblistes (partie de T∗T^\astT∗)

Théorème : Ce type d'inéquation admet une plus petite solution.

Idée de preuve : Soit F(exp)=⋃parties droite de reˋgleF(exp) = \bigcup \text{parties droite de règle}F(exp)=⋃parties droite de reˋgle

{F:Pn(T∗)→P(T∗)}i∈[1,n]\{F: \mathcal{P}^n(T^\ast) \rightarrow \mathcal{P}(T^\ast)\}_{i \in [1,n]}{F:Pn(T∗)→P(T∗)}i∈[1,n]​ avec nnn le nombre de terminaux. Exemple d'utilisation de cette grammaire​

Analyse syntaxique de 2×x+y2 \times x + y2×x+y, l'analyseur va reconnaître 2 comme une constance, ×\times× comme , xxx comme un ID, +++ comme + et yyy comme un ID. Comment pouvons nous vérifier que cette expression appartient bien au langage. Vérifions que csteID+ID est bien une entrée syntaxique correcte selon la grammaire.

Suite de dérivations (gauche) de exp→cste×ID+IDexp \rightarrow cste \times ID + IDexp→cste×ID+ID​

Remarques​

  • Dés qu'un terminal est généré ou engendré par une règle, on ne peut plus l'effacer
  • il coupe implicitement l'entrée en ce qui se trouve à gauche et ce qui se trouve à droite.

En pendant compilation, on peut avoir l'impression que cette suite de dérivation induit syntaxiquement une "priorité" sur les opérateurs binaires + et *

CSTE×ID⏟exp+⏟expID⏟exp\underbrace{CSTE \times ID}{exp} \underbrace{+}{exp} \underbrace{ID}{exp}CSTE×ID​exp+​expID​exp

Est ce satisfaisant ? Presque ... Expliciter la syntaxe, c'est bien. Mais, cette grammaire est ambigüe. Il y a au moins deux syntaxes possibles pour CST * ID + ID.

Cela est toujours une suite de dérivation gauche, mais la structure syntaxique induite est plutôt CSTE⏟exp×ID+ID⏟exp⏞parentheˋses implicites\underbrace{CSTE}{exp} \times \overbrace{\underbrace{ID + ID}_{exp}}^{\text{parenthèses implicites}}CSTE​exp×expID+ID​​​parentheˋses implicites​

Définition : Une grammaire est dites ambigüe lorsqu'il existe deux suites de dérivations gauches, distinctes, à partir du start symbol, qui produisent la même entrée.

En compilation, on ne veut pas ça ! En effet on va faire de la traduction dirigée par la syntaxe. Deux syntaxes possibles donneront deux codes différents. Exemple (2×x)+y≠2×(x+y)(2 \times x) + y \neq 2 \times (x + y)(2×x)+y=2×(x+y). En compilation, on n'utilise que des grammaires non ambigüe.

Arbre de dérivations​

Sur le même exemple 2×x+y2 \times x + y2×x+y

Ceci est un arbre de dérivation, aussi appelé arbre de syntaxe concret. Autrement, un arbre de dérivations c'est un arbre dont :

  • les sommets sont étiquetés sur TON
  • les embranchements définis par des règles de productions
  • les feuilles étiquetées sur T
  • la racine étiquetée par le start symbol

Remarque​

Il y a bijection entre arbre de dérivation et suites de dérivations gauches. Les suites de dérivations gauches sont exactement les parcours en profondeur, à gauche d'abord préfixe, des arbres de dérivations.

Essayer de construire un arbre de dérivation pour (IDID+)∉exp(ID ID +) \notin exp(IDID+)∈/exp est impossible

Définition : Une grammaire est ambigüe lorsque il existe deux arbres de dérivations pour une même entrée (un même mot de terminaux).

La grammaire ci-dessus quoique ambigüe, a tout de même le mérite de définir correctement les expressions arithmétiques. Cependant, ces expressions peuvent-elle être définies par une grammaire non ambigüe ?

Une première grammaire non ambigüe​

Idée : distinguer dans les expressions arithmétiques les termes qui composent les sommes de facteurs qui composent les produits. Autrement dit nous allons utiliser 3 noms terminaux.

exp→expressions arithmeˊtiquesexp \rightarrow \text{expressions arithmétiques}exp→expressions arithmeˊtiques term→termes dans une sommeterm \rightarrow \text{termes dans une somme}term→termes dans une somme fact→facteurs dans un produitfact \rightarrow \text{facteurs dans un produit}fact→facteurs dans un produit

N={exp,term,fact}N = \{exp,term,fact\}N={exp,term,fact}exp

Avec expexpexp comme start symbol. Les règles de cette grammaire :

  1. exp -> term
  2. exp -> exp + term
  3. term -> fact
  4. term -> term * fact
  5. fact -> ID
  6. fact -> CSTE
  7. fact -> (exp)

Pas d'ambiguité ? Nous voulons toujours dériver 2×x+y2 \times x + y2×x+y ie CSTE * ID + ID.

Cet arbre est le seul arbre de dérivation possible ! Le parenthésage induit par cet arbre est bien (2×x)+y(2 \times x) + y(2×x)+y. Nous avons donc notre grammaire pour la compilation. La grammaire utilisée s'appelle la grammaire ETF. Autre grammaire ?

  1. E→TE′E \rightarrow TE'E→TE′
  2. E′→ϵE' \rightarrow \epsilonE′→ϵ //mot vide
  3. E′→+TE′E' \rightarrow +TE'E′→+TE′
  4. T→FT′T \rightarrow FT'T→FT′
  5. T′→ϵ T' \rightarrow \epsilonT′→ϵ //mot vide
  6. T′→+FT′ T' \rightarrow +FT'T′→+FT′
  7. F→IDF \rightarrow IDF→ID
  8. F→CSTEF \rightarrow CSTEF→CSTE
  9. F→(E)F \rightarrow (E)F→(E)

On peut vérifier que cette grammaire engendre aussi les mêmes expressions arithmétiques et qu'elle n'est pas ambigüe.
